How it works

A deep dive into the RAG pipeline and evaluation methodology

This service implements Retrieval-Augmented Generation (RAG) — a pattern that grounds LLM answers in a verified document corpus rather than parametric memory alone. At query time, relevant chunks are retrieved from a vector store and injected into the prompt, giving the model authoritative jurisdiction-specific context it can cite directly.

Lewis et al. 2020 · arXiv:2005.11401
1

Document ingestion & chunking

Markdown employment-law documents are split into overlapping chunks (CHUNK_SIZE tokens, CHUNK_OVERLAP stride) and stored in a ChromaDB vector collection. Each chunk carries a doc_id and jurisdiction metadata field used for filtered retrieval.

chunk_i = doc[i·stride : i·stride + C],  stride = C − overlap
2

Dense embedding

Both document chunks and user queries are mapped to the same dense vector space using a sentence-transformer model (default: all-MiniLM-L6-v2). Semantic similarity in this space correlates with topical relevance, enabling retrieval far beyond keyword overlap.

Reimers & Gurevych 2019 · arXiv:1908.10084 Wang et al. 2020 · arXiv:2002.10957
3

Approximate nearest-neighbour retrieval

ChromaDB uses HNSW (Hierarchical Navigable Small World) graphs to find the top-k chunks closest to the query embedding in sub-linear time. An optional jurisdiction metadata filter narrows the search space before the ANN scan.

retrieve(q) = argmax_{c ∈ C_j} cos(embed(q), embed(c)),  |result| = k
Malkov & Yashunin 2018 · arXiv:1603.09320
4

Prompted generation

The retrieved chunks are assembled into a structured prompt (system prompt + context block + question block) and sent to the configured LLM (Ollama, OpenAI, or Anthropic). When LLM_PROVIDER=anthropic and ENABLE_PROMPT_CACHING=true, the system prompt and context block carry cache_control: ephemeral — repeated queries against the same corpus skip re-encoding those tokens, reducing latency and cost.

Lewis et al. 2020 · arXiv:2005.11401

Every run of the golden Q&A set measures five complementary signals across retrieval quality, generation faithfulness, and latency. The judge-based metrics use an LLM-as-evaluator pattern from the RAGAS framework — a structured prompt elicits a numeric score and chain-of-thought reasoning that can be inspected per question.

Es et al. 2023 · arXiv:2309.15217

Precision@k

Of the k chunks returned by the retriever, what fraction contain the expected document? Precision@k rewards the retriever for ranking relevant chunks highly without penalising recall.

Precision@k = |{relevant docs} ∩ {top-k retrieved}| / k
Lewis et al. 2020 · arXiv:2005.11401

Hit rate@k

A binary signal: did at least one of the top-k retrieved chunks come from the expected document? Hit@k is less strict than Precision@k and is a good proxy for whether the answer will be grounded at all.

Hit@k = 𝟙[∃ c ∈ top-k : doc(c) = doc_expected]
Es et al. 2023 · arXiv:2309.15217

Faithfulness

An LLM judge reads the generated answer alongside the retrieved context and decides whether every factual claim in the answer is supported by that context. When the provider is Anthropic, the judge uses tool use for structured output; otherwise a JSON-parsing fallback is used.

faithfulness_score ∈ [0, 1],  1 = fully supported by context
Zheng et al. 2023 · arXiv:2306.05685 Es et al. 2023 · arXiv:2309.15217

Context relevance

A RAGAS-style metric: the judge estimates what fraction of sentences in the retrieved context are actually needed to answer the question. Low context relevance means the retriever is returning noisy or tangential chunks that inflate the prompt without aiding generation.

context_relevance = |{sentences needed}| / |{sentences in context}|  ∈ [0, 1]
Es et al. 2023 · arXiv:2309.15217

p95 latency

The 95th-percentile end-to-end query latency across the golden set, measured in milliseconds. p95 rather than mean is reported because tail latency determines user-visible worst-case experience; outliers caused by cold cache misses or LLM queuing are surfaced here rather than averaged away.

p95 = percentile(latencies, 95),  target ≤ 1 000 ms
